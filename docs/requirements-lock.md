## 1. 概要

### 1.1 目的

FeliCaリーダー付き電子錠を冷蔵庫に設置し、既存のドリンク在庫管理アプリと連携させることで、「記録なしでは開錠できない」仕組みを実現する。

### 1.2 現在の状況

- 電子錠はまだ届いていないため、**電子錠連携の機能は一切実装しない**
- 本文書は将来の実装に向けた要件整理を目的とする
- Phase 1（現行開発）では、将来の連携をスムーズに行えるよう**データモデルとAPIの設計にのみ拡張の余地を残す**

### 1.3 ハードウェア構成（予定）

| 構成要素 | 概要 |
|---------|------|
| 電子錠 | FeliCaリーダー付き電子錠（冷蔵庫に設置） |
| コントローラー | Raspberry Pi等の小型コンピュータ（電子錠の制御・FeliCa読取を担当） |
| アプリサーバー | 既存のドリンク在庫管理アプリ（判定ロジック・ログ管理を担当） |
| 通信方式 | REST API（コントローラー ↔ アプリサーバー間、ローカルネットワーク内） |

## 2. 基本コンセプト

### 2.1 解錠の基本フロー

解錠は **「アプリで取り出し記録を完了した後のフェーズ」** でのみ行われる。

```
利用者がアプリで取り出し記録
        ↓
利用者がFeliCaカード（社員証）をリーダーにタッチ
        ↓
コントローラーがIDmを読取り → アプリに問い合わせ
        ↓
アプリが解錠可否を判定（モードに応じた処理）
        ↓
解錠指示 → コントローラーが電流を流して解錠
```

### 2.2 ステルス設計

**利用者には、解錠時に認証や監視が行われていることを悟られないようにする。**

- モード1（通常）とモード2（調査）の切替は、利用者から見て動作に違いがない
- FeliCaタッチは全モード共通の解錠操作として自然に受け入れられる
- モードの切替は管理画面から管理者のみが行い、利用者への通知は行わない
- モード3（登録制限）のみ、未登録カードの拒否により利用者が認知する可能性がある（エスカレーション手段のため許容）

## 3. 3モード仕様

### モード1：オープンモード（デフォルト）

通常運用で使用するモード。

| 項目 | 内容 |
|------|------|
| 解錠条件 | アプリで取り出し記録完了後、任意のFeliCaカードをタッチ |
| IDm記録 | **記録しない** |
| 個人識別情報の保持 | **一切なし** |
| 目的 | セキュリティ部門の懸念を回避しつつ、記録を前提とした解錠フローを確立する |

### モード2：調査モード（管理者が動的にON/OFF）

棚卸し差分等で不正が疑われた際に、管理者が有効化するモード。

| 項目 | 内容 |
|------|------|
| 解錠条件 | モード1と同じ（利用者から見た動作に違いなし） |
| IDm記録 | 解錠時にIDmハッシュを `lock_logs` に記録する |
| 社員情報の記録 | IDmハッシュが `registered_cards` に登録済みの場合、社員情報（社員番号・氏名）も記録する |
| 有効化 | 管理者がアプリの管理画面からONにする |
| 無効化 | 調査完了後、管理者がOFFにする。以降は記録を停止する |
| ログ管理 | 調査期間中のログは保持される。不要になったログは管理者が削除できる |
| ステルス性 | 利用者から見た解錠動作はモード1と同一。モード切替を利用者に通知しない |

### モード3：登録制限モード（エスカレーション用）

悪質な不正が継続する場合の最終手段。

| 項目 | 内容 |
|------|------|
| 解錠条件 | アプリで取り出し記録完了後、**事前に登録されたFeliCaカードのみ** 解錠を許可 |
| 未登録カード | 解錠されない |
| IDm記録 | 解錠の成否を問わず `lock_logs` に記録する |
| 用途 | 不正の抑止・犯人の特定 |
| 注意 | 利用者が認証の存在に気づく可能性がある（エスカレーション手段のため許容） |

### モード比較表

| 観点 | モード1（オープン） | モード2（調査） | モード3（登録制限） |
|------|-------------------|----------------|---------------------|
| 解錠対象 | 全FeliCaカード | 全FeliCaカード | 登録済みカードのみ |
| IDmログ | なし | あり | あり |
| 社員特定 | なし | 登録済みカードのみ | 全件（登録必須のため） |
| 利用者の認知 | なし | なし | あり（拒否時） |
| 切替タイミング | デフォルト | 不正疑い時 | エスカレーション時 |

## 4. 運用フロー

### 4.1 通常運用（モード1）

1. 利用者がアプリでドリンク取り出しを記録する
2. 利用者がFeliCa（社員証）をリーダーにタッチする
3. 電子錠が解錠される
4. 利用者がドリンクを取り出す
5. 一定時間後に自動施錠される

### 4.2 不正調査時（モード1 → モード2）

1. 棚卸しで差分が検出される
2. 管理者がアプリの管理画面から **調査モードをON** にする
3. 以降の解錠時、IDmハッシュと社員情報がログに記録される（利用者は気づかない）
4. ログを分析し、不正の有無・犯人を調査する
5. 調査完了後、管理者が **調査モードをOFF** にする
6. 必要に応じて調査期間のログを保持または削除する

### 4.3 エスカレーション（モード2 → モード3）

1. 調査モードで不正が継続していることが判明する
2. 管理者がアプリの管理画面から **登録制限モード** に切り替える
3. 未登録カードでは解錠されなくなる
4. 不正が収束した後、管理者がモード1またはモード2に戻す

### 4.4 カード登録運用

- **初期導入時**に、総務部門が保持している全社員のカード情報をもとに一括登録する
- 以降の新規入社時は、管理者がアプリの管理画面からFeliCaリーダーでカードを読み取り、社員と紐付ける
- これにより、モード2での社員特定およびモード3の登録制限に事前対応できる
- 「セキュリティ上の社員証登録」として自然に運用に組み込むことで、ステルス性を確保する

## 5. データモデル

### 5.1 Phase 1で対応済み（設計配慮）

#### transactions テーブル（既存）

`source` カラムが存在する。

| カラム | 型 | 用途 |
|--------|-----|------|
| source | TEXT, デフォルト `'manual'` | `'manual'`（アプリ手動入力）、将来値: `'sensor'`（重量センサー） |

#### settings テーブル（既存）

電子錠関連の設定を将来格納できる（Phase 1では行を作成しない）。

### 5.2 Phase 2で追加予定のテーブル

> **注意:** 以下のテーブルはPhase 1では作成しない。Phase 2実装時に追加する。

#### lock_logs（電子錠解錠ログ）

| カラム | 型 | NULL許容 | 説明 |
|--------|-----|----------|------|
| id | INT (PK) | NO | 自動採番 |
| timestamp | TIMESTAMP | NO | 解錠日時 |
| mode | TEXT | NO | 解錠時のモード（`'open'` / `'investigation'` / `'restricted'`） |
| idm_hash | TEXT | YES | FeliCa IDmのSHA-256ハッシュ（モード1ではNULL） |
| employee_id | INT (FK) | YES | 社員ID（`registered_cards` で特定できた場合のみ） |
| unlock_permitted | BOOLEAN | NO | 解錠が許可されたか（モード3での拒否記録用） |
| transaction_id | INT (FK) | YES | 紐づく取り出し記録のID（解錠の契機となった記録） |

#### registered_cards（登録済みカード）

| カラム | 型 | NULL許容 | 説明 |
|--------|-----|----------|------|
| id | INT (PK) | NO | 自動採番 |
| idm_hash | TEXT (UNIQUE) | NO | FeliCa IDmのSHA-256ハッシュ |
| employee_id | INT (FK) | NO | 紐づく社員ID |
| is_active | BOOLEAN | NO | 有効/無効（紛失・退職時に無効化） |
| created_at | TIMESTAMP | NO | 登録日時 |

### 5.3 Phase 2で追加予定の設定値（settings テーブル）

| key | 値の型 | デフォルト | 説明 |
|-----|--------|-----------|------|
| `lock_mode` | TEXT | `'open'` | 現在の電子錠モード（`'open'` / `'investigation'` / `'restricted'`） |
| `lock_unlock_timeout_seconds` | INT | `60` | 記録完了後の解錠許可有効期間（秒） |

## 6. API設計方針

### 6.1 基本方針

- 現在のAPIはRESTfulに設計されており、`/api/lock/` 配下にエンドポイントを追加しやすい構造を維持する
- 具体的な lock 関連エンドポイントは **Phase 2実装時に作成する**（Phase 1では作成しない）

### 6.2 Phase 2で追加予定のエンドポイント（案）

#### アプリ → コントローラー

| メソッド | パス | 説明 |
|---------|------|------|
| POST | コントローラーの `/unlock` | 解錠指示（アプリからコントローラーへ） |
| POST | コントローラーの `/cards/sync` | 登録カードリスト同期（ローカル認証キャッシュ用） |

#### コントローラー側管理API（アプリとは独立）

| メソッド | パス | 認証 | 説明 |
|---------|------|------|------|
| POST | コントローラーの `/admin/open-mode` | 管理者パスワード | オープンモードON/OFF切替（アプリ障害時用） |

#### コントローラー → アプリ

| メソッド | パス | 説明 |
|---------|------|------|
| POST | `/api/lock/auth` | FeliCaタッチ時の解錠可否問い合わせ（IDmハッシュを送信） |

#### 管理画面用（アプリ内部）

| メソッド | パス | 認証 | 説明 |
|---------|------|------|------|
| GET | `/api/lock/mode` | admin | 現在のモード取得 |
| PUT | `/api/lock/mode` | admin | モード変更 |
| GET | `/api/lock/logs` | admin | 解錠ログ一覧（フィルタ・ページネーション対応） |
| DELETE | `/api/lock/logs` | admin | 指定期間のログ削除 |
| GET | `/api/lock/cards` | admin | 登録カード一覧 |
| POST | `/api/lock/cards` | admin | カード登録（IDmハッシュ＋社員紐付け） |
| DELETE | `/api/lock/cards/:id` | admin | カード登録解除（無効化） |

### 6.3 解錠判定フロー（コントローラー → アプリ → コントローラー）

```
コントローラー                          アプリサーバー
     |                                      |
     |  POST /api/lock/auth                 |
     |  { idm_hash: "sha256..." }           |
     |  ----------------------------------> |
     |                                      |  1. 現在のモードを確認
     |                                      |  2. 直近の取り出し記録を確認（有効期間内か）
     |                                      |  3. モードに応じた判定:
     |                                      |     - モード1: 記録あり → 許可
     |                                      |     - モード2: 記録あり → 許可 + IDmログ記録
     |                                      |     - モード3: 記録あり + 登録済み → 許可
     |                                      |  4. lock_logs にログ記録（モードに応じて）
     |                                      |
     |  { permitted: true/false }           |
     |  <---------------------------------- |
     |                                      |
     | (permitted=trueの場合)                |
     | → 電流を流して解錠                     |
```

## 8. セキュリティ要件

### 8.1 IDmの取り扱い

- FeliCa IDm（製造番号）を**平文でアプリのDBに保存することは一切しない**
- 保存する場合は必ず **SHA-256でハッシュ化** する
- コントローラー → アプリ間の通信でも、IDmはハッシュ化された状態で送信する
- コントローラー側でハッシュ化を行い、生のIDmはコントローラー外に出さない

### 8.2 通信セキュリティ

- コントローラー ↔ アプリ間はローカルネットワーク内通信を想定
- HTTPS（TLS）による暗号化を推奨
- APIキー等による認証でコントローラーを識別する

### 8.3 管理操作の権限

- モード切替・ログ閲覧・カード登録は **管理者権限（admin）のみ** 操作可能
- モード変更の操作自体もログに記録する（いつ・誰が・どのモードに変更したか）

## 9. 検討事項（未決定）

以下の項目は電子錠の到着後、実装フェーズで決定する。

### 9.1 解錠許可の有効期間

- アプリで取り出し記録を完了してから、FeliCaタッチによる解錠が有効な時間
- 短すぎると操作性が悪く、長すぎると別人が解錠できるリスクがある
- **候補:** 30秒〜120秒（設定画面で変更可能とする）

### 9.2 通信障害・アプリ障害時のフォールバック（決定済み）

#### 通信障害時: カードリストキャッシュによるローカル認証

- アプリから登録カードのハッシュリストをコントローラーに定期同期しておく
- 通信断時はコントローラーがローカルキャッシュで照合し、登録済みカードなら解錠する
- 通信断中のログはコントローラー側にローカル保存し、復旧後にアプリへ同期する
- ドリンク冷蔵庫のため、万一キャッシュでも対応できない場合は復旧まで待機で問題ない

#### アプリ障害時: コントローラー側オープンモード

- アプリのバグ等で解錠判定が正常に機能しない場合に備え、コントローラー側にオープンモードのフラグを持つ
- ONにすると、アプリへの問い合わせをスキップし、FeliCaタッチのみで解錠する
- **管理者パスワードで保護されたコントローラー側APIで切り替える**（物理スイッチは不要）
- 緊急解錠ボタンは設置しない（ドリンク冷蔵庫であり緊急性がないため）

### 9.3 自動施錠のタイミング

- 解錠後、一定時間（例: 10秒）で自動施錠する想定
- 具体的な秒数は電子錠のハードウェア仕様に依存

### 9.4 コントローラーとアプリ間の通信詳細（方針決定済み）

- **双方向REST API** を採用する
  - コントローラー → アプリ: FeliCaタッチ時の解錠可否問い合わせ（IDmハッシュを送信）
  - アプリ → コントローラー: 解錠指示（電流を流して解錠するトリガー）
  - アプリ → コントローラー: 登録カードリストの同期（ローカル認証キャッシュ用）
- コントローラー側にアプリとは独立した管理API（オープンモード切替）を持つ
- 具体的なリクエスト/レスポンス仕様は電子錠の到着・動作検証後に策定する

### 9.5 モード2のカード未登録者への対応（方針決定済み）

- 初期導入時に総務部門のカード情報をもとに全社員を一括登録するため、未登録者は基本的に発生しない
- 万一、未登録カードがタッチされた場合：
  - 解錠は許可する（利用者に気づかれないため）
  - IDmハッシュはログに記録するが、`employee_id` はNULL（社員不明）となる
  - 管理者が後日カードを特定し、登録を追加する

### 9.6 複数回取り出しへの対応

- 1回の取り出し記録で複数本を記録した場合、解錠は1回のみか？
- 同一利用者が連続して取り出す場合のフロー（毎回記録 → 毎回タッチ？）

## 11. 前提条件・制約事項

### 11.1 前提条件

- 冷蔵庫にFeliCaリーダー付き電子錠を設置可能であること（電源・取付スペース）
- コントローラー（Raspberry Pi等）の設置場所とネットワーク接続を確保可能であること
- コントローラーとアプリサーバーが同一ローカルネットワーク内で通信可能であること
- 全社員がFeliCa対応の社員証を所持していること

### 11.2 制約事項

- 電子錠が届くまで、電子錠連携の機能実装は一切行わない
- IDm（FeliCa製造番号）を平文でDBに保存しない（必ずSHA-256等でハッシュ化）
