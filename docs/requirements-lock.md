## 1. 概要

### 1.1 目的

電子錠（スナッチロック）とPaSoRi（FeliCaリーダー）を冷蔵庫付近に設置し、既存のドリンク在庫管理アプリと連携させることで、「記録なしでは開錠できない」仕組みを実現する。

### 1.2 現在の状況

- 電子錠はまだ届いていないため、**電子錠連携の機能は一切実装しない**
- 本文書は将来の実装に向けた要件整理を目的とする
- Phase 1（現行開発）では、将来の連携をスムーズに行えるよう**データモデルとAPIの設計にのみ拡張の余地を残す**

### 1.3 ハードウェア構成（予定）

| 構成要素 | 設置場所 | 概要 |
|---------|---------|------|
| iPad（タブレット） | 冷蔵庫前に固定設置 | ブラウザでWebアプリを表示するだけ。ハードウェアは接続しない |
| Raspberry Pi | 冷蔵庫付近 | **アプリサーバー + PaSoRi + 電子錠制御をすべて1台で担当** |
| PaSoRi（FeliCaリーダー） | Raspberry PiにUSB接続 | ICカードのIDm読み取りを担当（Sony RC-S300等） |
| 電子錠 | 冷蔵庫 | タキゲン スナッチロック（LE-31 / LE-35等、未選定）。Raspberry PiのGPIO経由で通電制御 |

#### 構成図

```
[iPad]                          [Raspberry Pi]
  ブラウザ ── Wi-Fi/LAN ──→  Next.js アプリサーバー
  （表示のみ）                      │
                                ├── PaSoRi (USB) ← FeliCaカード読み取り
                                └── 電子錠 (GPIO + リレー/MOSFET) ← 施錠/解錠制御
```

> **ポイント:** アプリサーバーと電子錠制御が同一の Raspberry Pi 上で動作するため、両者間のネットワーク通信は不要。PaSoRi もこの Pi に直接接続される。iPadはブラウザとしてのみ機能する。

## 2. 基本コンセプト

### 2.1 解錠の基本フロー

解錠は **「アプリで取り出し記録を完了した後のフェーズ」** でのみ行われる。

```
利用者がiPadのブラウザでドリンク取り出しを記録
        ↓
利用者がFeliCaカード（社員証）をRaspberry Pi上のPaSoRiにタッチ
        ↓
Raspberry PiがPaSoRi経由でIDmを読取り
        ↓
Raspberry Pi上のアプリが解錠可否を判定（モードに応じた処理）
        ↓
Raspberry PiがGPIOで電子錠に通電して解錠
```

> **ポイント:** カード読み取り・解錠判定・電子錠制御がすべてRaspberry Pi 1台で完結する。iPadはブラウザUIの表示のみ。

### 2.2 ステルス設計

**利用者には、解錠時に認証や監視が行われていることを悟られないようにする。**

- モード1（通常）とモード2（調査）の切替は、利用者から見て動作に違いがない
- 冷蔵庫横のPaSoRiへのFeliCaタッチは全モード共通の解錠操作として自然に受け入れられる
- モードの切替は管理画面から管理者のみが行い、利用者への通知は行わない
- モード3（登録制限）のみ、未登録カードの拒否により利用者が認知する可能性がある（エスカレーション手段のため許容）

## 3. 3モード仕様

### モード1：オープンモード（デフォルト）

通常運用で使用するモード。

| 項目 | 内容 |
|------|------|
| 解錠条件 | アプリで取り出し記録完了後、任意のFeliCaカードをタッチ |
| IDm記録 | **記録しない** |
| 個人識別情報の保持 | **一切なし** |
| 目的 | セキュリティ部門の懸念を回避しつつ、記録を前提とした解錠フローを確立する |

### モード2：調査モード（管理者が動的にON/OFF）

棚卸し差分等で不正が疑われた際に、管理者が有効化するモード。

| 項目 | 内容 |
|------|------|
| 解錠条件 | モード1と同じ（利用者から見た動作に違いなし） |
| IDm記録 | 解錠時にIDmハッシュを `lock_logs` に記録する |
| 社員情報の記録 | IDmハッシュが `registered_cards` に登録済みの場合、社員情報（社員番号・氏名）も記録する |
| 有効化 | 管理者がアプリの管理画面からONにする |
| 無効化 | 調査完了後、管理者がOFFにする。以降は記録を停止する |
| ログ管理 | 調査期間中のログは保持される。不要になったログは管理者が削除できる |
| ステルス性 | 利用者から見た解錠動作はモード1と同一。モード切替を利用者に通知しない |

### モード3：登録制限モード（エスカレーション用）

悪質な不正が継続する場合の最終手段。

| 項目 | 内容 |
|------|------|
| 解錠条件 | アプリで取り出し記録完了後、**事前に登録されたFeliCaカードのみ** 解錠を許可 |
| 未登録カード | 解錠されない |
| IDm記録 | 解錠の成否を問わず `lock_logs` に記録する |
| 用途 | 不正の抑止・犯人の特定 |
| 注意 | 利用者が認証の存在に気づく可能性がある（エスカレーション手段のため許容） |

### モード比較表

| 観点 | モード1（オープン） | モード2（調査） | モード3（登録制限） |
|------|-------------------|----------------|---------------------|
| 解錠対象 | 全FeliCaカード | 全FeliCaカード | 登録済みカードのみ |
| IDmログ | なし | あり | あり |
| 社員特定 | なし | 登録済みカードのみ | 全件（登録必須のため） |
| 利用者の認知 | なし | なし | あり（拒否時） |
| 切替タイミング | デフォルト | 不正疑い時 | エスカレーション時 |

## 4. 運用フロー

### 4.1 通常運用（モード1）

1. 利用者がiPadのブラウザでドリンク取り出しを記録する
2. 利用者がFeliCa（社員証）をRaspberry Pi上のPaSoRiにタッチする
3. Raspberry Pi上のアプリがIDmを読み取り、解錠判定を行う
4. 同一Pi上でGPIO経由で電子錠に通電し、解錠される
5. 利用者がドリンクを取り出す
6. 一定時間後に自動施錠される

### 4.2 不正調査時（モード1 → モード2）

1. 棚卸しで差分が検出される
2. 管理者がアプリの管理画面から **調査モードをON** にする
3. 以降の解錠時、Raspberry Pi上のPaSoRiで読み取ったIDmハッシュと社員情報がログに記録される（利用者は気づかない）
4. ログを分析し、不正の有無・犯人を調査する
5. 調査完了後、管理者が **調査モードをOFF** にする
6. 必要に応じて調査期間のログを保持または削除する

### 4.3 エスカレーション（モード2 → モード3）

1. 調査モードで不正が継続していることが判明する
2. 管理者がアプリの管理画面から **登録制限モード** に切り替える
3. 未登録カードでは解錠されなくなる
4. 不正が収束した後、管理者がモード1またはモード2に戻す

### 4.4 カード登録運用

- **初期導入時**に、総務部門が保持している全社員のカード情報をもとに一括登録する
- 以降の新規入社時は、管理者がアプリの管理画面からPaSoRiでカードを読み取り、社員と紐付ける
- これにより、モード2での社員特定およびモード3の登録制限に事前対応できる
- 「セキュリティ上の社員証登録」として自然に運用に組み込むことで、ステルス性を確保する

## 6. API設計方針

### 6.1 基本方針

- アプリサーバーと電子錠制御が同一Raspberry Pi上で動作するため、**デバイス間のネットワークAPIは不要**
- アプリ（Next.js）から電子錠制御・PaSoRi読み取りは、同一プロセス内の関数呼び出しまたはローカルIPC（プロセス間通信）で行う
- 現在のAPIはRESTfulに設計されており、`/api/lock/` 配下に管理画面用エンドポイントを追加しやすい構造を維持する
- 具体的な lock 関連エンドポイントは **Phase 2実装時に作成する**（Phase 1では作成しない）

### 6.2 Phase 2で追加予定のエンドポイント（案）

> **構成のポイント:** PaSoRi・電子錠・アプリサーバーがすべて同一Raspberry Pi上にあるため、iPad（ブラウザ）向けの管理APIのみを定義する。デバイス間通信APIは不要。

#### 管理画面用（iPad → Raspberry Pi、既存のNext.js APIに追加）

| メソッド | パス | 認証 | 説明 |
|---------|------|------|------|
| GET | `/api/lock/mode` | admin | 現在のモード取得 |
| PUT | `/api/lock/mode` | admin | モード変更 |
| GET | `/api/lock/logs` | admin | 解錠ログ一覧（フィルタ・ページネーション対応） |
| DELETE | `/api/lock/logs` | admin | 指定期間のログ削除 |
| GET | `/api/lock/cards` | admin | 登録カード一覧 |
| POST | `/api/lock/cards` | admin | カード登録（IDmハッシュ＋社員紐付け） |
| DELETE | `/api/lock/cards/:id` | admin | カード登録解除（無効化） |

### 6.3 解錠判定フロー（すべてRaspberry Pi上で完結）

```
iPad（ブラウザ）           Raspberry Pi（アプリ + PaSoRi + GPIO）
     |                          |
     | 取り出し記録を送信        |
     | -----------------------> |  記録をDBに保存
     |                          |
     |                          |  ── PaSoRi でカード待受中 ──
     |                          |
     |                          |  FeliCaタッチ検知 → IDm読取り
     |                          |
     |                          |  1. 現在のモードを確認
     |                          |  2. 直近の取り出し記録を確認（有効期間内か）
     |                          |  3. モードに応じた判定:
     |                          |     - モード1: 記録あり → 許可
     |                          |     - モード2: 記録あり → 許可 + IDmログ記録
     |                          |     - モード3: 記録あり + 登録済み → 許可
     |                          |  4. lock_logs にログ記録
     |                          |
     |                          |  (許可の場合)
     |                          |  → GPIO で電子錠に通電して解錠
     |                          |
     |  解錠結果を画面に反映     |
     | <----------------------- |
```

> **構成のメリット:** PaSoRi・解錠判定・電子錠制御がすべて同一デバイス上にあるため、ネットワーク遅延やデバイス間通信障害のリスクがない。

## 8. セキュリティ要件

### 8.1 IDmの取り扱い

- FeliCa IDm（製造番号）を**平文でDBに保存することは一切しない**
- 保存する場合は必ず **SHA-256でハッシュ化** する
- PaSoRiから読み取ったIDmは、Raspberry Pi上のアプリ内で即座にハッシュ化し、生のIDmはメモリ上にのみ一時保持する

### 8.2 通信セキュリティ

- iPad ↔ Raspberry Pi間はローカルネットワーク内通信を想定
- HTTPS（TLS）による暗号化を推奨
- アプリサーバーと電子錠制御は同一デバイス上のため、デバイス間通信のセキュリティ考慮は不要

### 8.3 管理操作の権限

- モード切替・ログ閲覧・カード登録は **管理者権限（admin）のみ** 操作可能
- モード変更の操作自体もログに記録する（いつ・誰が・どのモードに変更したか）

## 9. 検討事項（未決定）

以下の項目は電子錠の到着後、実装フェーズで決定する。

### 9.1 解錠許可の有効期間

- アプリで取り出し記録を完了してから、FeliCaタッチによる解錠が有効な時間
- 短すぎると操作性が悪く、長すぎると別人が解錠できるリスクがある
- **候補:** 30秒〜120秒（設定画面で変更可能とする）

### 9.2 アプリ障害時のフォールバック（方針決定済み）

> **構成上の簡素化:** アプリサーバーと電子錠制御が同一Raspberry Pi上にあるため、従来の「アプリ↔コントローラー間の通信障害」は発生しない。障害パターンはRaspberry Pi自体の故障・停止のみとなる。

#### Raspberry Pi停止時

- アプリも電子錠制御も同一デバイスのため、Pi停止時は解錠不可
- ドリンク冷蔵庫であり緊急性がないため、復旧まで待機で問題ない
- 緊急解錠ボタンは設置しない

#### アプリプロセス障害時

- アプリのバグ等で解錠判定が正常に機能しない場合に備え、電子錠制御プロセスに独立したフォールバックモードを持たせることを検討
- フォールバックモードON時は、PaSoRiへの任意のカードタッチで解錠を許可する
- 具体的な実装方式（watchdog、別プロセス等）はPhase 2実装時に決定する

### 9.3 自動施錠のタイミング

- 解錠後、一定時間（例: 10秒）で自動施錠する想定
- 具体的な秒数は電子錠のハードウェア仕様に依存

### 9.4 アプリと電子錠制御の連携方式（方針決定済み）

- アプリサーバーと電子錠制御が同一Raspberry Pi上にあるため、**デバイス間のネットワークAPIは不要**
- 連携方式の候補:
  - **同一プロセス:** Next.jsアプリ内からNode.jsのGPIOライブラリ（`onoff`等）とNFCライブラリ（`nfc-pcsc`等）を直接呼び出す
  - **別プロセス:** 電子錠制御・PaSoRi読み取りをデーモンプロセスとして分離し、ローカルIPC（Unix Socket等）で通信する
- 具体的な方式は電子錠の到着・動作検証後に決定する

### 9.5 モード2のカード未登録者への対応（方針決定済み）

- 初期導入時に総務部門のカード情報をもとに全社員を一括登録するため、未登録者は基本的に発生しない
- 万一、未登録カードがタッチされた場合：
  - 解錠は許可する（利用者に気づかれないため）
  - IDmハッシュはログに記録するが、`employee_id` はNULL（社員不明）となる
  - 管理者が後日カードを特定し、登録を追加する

### 9.6 複数回取り出しへの対応

- 1回の取り出し記録で複数本を記録した場合、解錠は1回のみか？
- 同一利用者が連続して取り出す場合のフロー（毎回記録 → 毎回タッチ？）

## 11. 前提条件・制約事項

### 11.1 前提条件

- 冷蔵庫にスナッチロック（LE-31 / LE-35等）を設置可能であること（電源・取付スペース）
- Raspberry Piの設置場所（冷蔵庫付近）と電源・ネットワーク接続を確保可能であること
- iPadとRaspberry Piが同一ローカルネットワーク内で通信可能であること
- 全社員がFeliCa対応の社員証を所持していること
- PaSoRi（Sony RC-S300等）がRaspberry PiのUSBポートに接続可能であること

### 11.2 制約事項

- 電子錠が届くまで、電子錠連携の機能実装は一切行わない
- IDm（FeliCa製造番号）を平文でDBに保存しない（必ずSHA-256等でハッシュ化）
- 電子錠のハードウェア仕様が未確定のため、GPIO制御の詳細はPhase 2開始時に策定する
- Phase 2導入時、アプリのデプロイ先をAWS Lightsail → Raspberry Piに移行する（iPad向けWebアプリをPiからホスティング）
