## 1. プロジェクト概要

### 1.1 背景・課題

現在、来客時のドリンク提供において、ガラススライド式冷蔵庫からの取り出し記録をGoogleフォームで管理しているが、以下の課題が発生している。

- **消費記録の入れ忘れ**: 個人の携帯端末からの入力のため、取り出し時に記録を忘れるケースが頻発
- **在庫数の不一致**: 理論在庫と実在庫に常に差分が発生し、正確な在庫把握ができない
- **無断持ち出し（盗難）の可能性**: 記録なしでの持ち出しを防止する手段がない

### 1.2 目的

ブラウザベースの在庫管理アプリケーションを開発し、冷蔵庫前に設置するタブレットで運用することで、記録の入れ忘れ防止および無断持ち出しの抑止を実現する。

### 1.3 解決アプローチ

本システムでは、冷蔵庫前に設置するタブレット端末からブラウザベースのアプリで取り出し記録を行う仕組みを構築する。取り出し動線上にタブレットを配置し、記録操作を簡素化することで入れ忘れを防止する。棚卸し機能と差分アラートにより、無断持ち出しの早期検知も実現する。

将来的には以下の拡張を予定している。

- **Phase 2**: ICカード対応電子錠を導入し、アプリと連携させることで「記録なしでは開錠できない」仕組みを実現
- **Phase 3**: 重量センサーを導入し、冷蔵庫内の在庫数を自動計測・記録する仕組みを実現

### 1.4 対象範囲

- 冷蔵庫：ガラススライド式冷蔵庫（1台）
- 端末：タブレット（冷蔵庫前に固定設置）
- 利用者：社内の来客対応スタッフ、管理者
- 【将来対応】鍵：ICカード対応電子錠（別途導入予定）
- 【将来対応】センサー：重量センサー（冷蔵庫内設置予定）

| ---- | ----------------------- | ------------------------------------------------------------------------------------------- |
| S-01 | ログイン画面                  | 8桁の社員番号の入力によりログインする画面。テンキーUI対応。パスワード入力は不要。                                                     |

| S-04 | 入庫登録画面                  | ドリンク補充時の入庫数量を登録する画面。管理者権限。                                                                  |
| S-05 | 棚卸し画面                   | ドリンク種類ごとの実在庫数を入力する画面。理論在庫との差分を自動表示。確定時に在庫数を実在庫に更新。結果サマリーを表示。                          |
| S-06 | ダッシュボード画面               | 在庫状況・本日の消費数・差分アラートをグラフとカードで表示する管理画面。                                                        |

| S-08 | マスタ管理画面                 | ドリンクマスタ・社員マスタの追加・編集・削除を行う画面。社員はCSV一括登録対応。管理者権限。                                             |
| S-09 | 設定画面                    | アラート通知先・在庫低下閾値・棚卸しスケジュール等のシステム設定画面。管理者権限。【将来】ICカード連携設定・重量センサー設定。                       |
| S-10 | 消費統計画面                  | 全ユーザー閲覧可能。社員別消費ランキング、ドリンク別消費割合（円グラフ）、在庫消費ペーステーブル、日別消費推移（棒グラフ・折れ線グラフ）。      |

### 3.1 メイン画面（S-02）の操作フロー

1. 社員番号を入力してログイン（S-01）
1. ドリンク種類を3列コンパクトリストから選択（S-02）
1. テンキー（1〜9）で数量を1タップで選択（在庫数を超えるボタンは無効化）
1. お客様名を入力（任意）
1. 「確定」ボタンを押下
1. 記録完了画面が表示される（S-03）
1. 冷蔵庫からドリンクを取り出す
1. 3秒後に自動でメイン画面に戻る

## 5. 技術構成

### 5.1 技術スタック

| 項目             | 技術                              | 選定理由                                                      |
| -------------- | ------------------------------- | --------------------------------------------------------- |
| 言語             | TypeScript                      | フロント・バックを1言語で統一。型安全性により品質確保。                              |
| フレームワーク        | Next.js（App Router）             | API Routes でバックエンドも内包。SSR/SSG対応。PWA化でオフライン記録も可能。          |
| ORM            | Prisma                          | TypeScript との親和性が高く、SQLite → PostgreSQL のDB切替が容易。         |
| データベース         | SQLite（本番）                      | 小規模運用に十分。サーバー内ファイルで完結しDB管理コスト不要。将来的にPostgreSQLへ移行可能。      |
| UIライブラリ        | Tailwind CSS + shadcn/ui        | タブレット向けの大きなタップ領域を実現しやすい。                                  |
| 認証             | 社員番号ベースの簡易認証（JWT）               | パスワード不要。外部認証サービス不要でコスト抑制。                                 |
| 通知             | メール通知（Nodemailer）またはSlack Webhook | 管理者向けアラート用。                                               |
| 【将来】センサー通信     | MQTT または WebSocket              | 重量センサーからのリアルタイムデータ受信。Node.jsエコシステムが充実。                    |
| 【将来】IoT基盤      | Raspberry Pi / ESP32            | センサーデータの中継デバイス。                                           |

### 5.2 開発環境（ローカル）

| 項目         | 内容                                          |
| ---------- | ------------------------------------------- |
| ランタイム      | Node.js 20 LTS 以上                           |
| パッケージマネージャ | npm または pnpm                                |
| データベース     | SQLite（ローカルファイル、インストール不要）                    |
| 起動コマンド     | `npm run dev`（Next.js開発サーバー、ホットリロード対応）       |
| 環境変数       | `.env.local` で管理（JWT秘密鍵、管理者初期パスワード等）         |
| DB初期化      | `npx prisma migrate dev` でスキーマ適用＋シードデータ投入    |

ローカル環境では Node.js のインストールのみで開発を開始できること。Docker は必須としない（任意で利用可能）。

### 5.3 本番環境（AWS）

| 項目      | 内容                                                              |
| ------- | --------------------------------------------------------------- |
| デプロイ先   | AWS Lightsail（最小インスタンス: $3.5/月〜）                                |
| OS      | Ubuntu 24.04 LTS                                                |
| プロセス管理  | PM2（Node.jsプロセスマネージャ）                                           |
| HTTPS   | Lightsail ロードバランサー（無料SSL証明書）または Let's Encrypt                   |
| ドメイン    | Lightsail 静的IP + 任意ドメイン（Route 53 または社内DNS）                      |
| バックアップ  | Lightsail 自動スナップショット（日次） + SQLiteファイルのS3バックアップ（日次cron）          |
| データベース  | SQLite（サーバー内ファイル）。将来スケールする場合は RDS（PostgreSQL）へ移行。               |
| 月額コスト目安 | $3.5〜$5（Lightsailインスタンス） + $0〜$1（S3バックアップ）= **約500〜900円/月**     |

### 5.4 デプロイ手順（概要）

1. GitHub リポジトリにプッシュ
1. Lightsail インスタンスにSSH接続
1. `git pull` → `npm install` → `npx prisma migrate deploy` → `npm run build`
1. PM2 でプロセス再起動（`pm2 restart all`）

将来的にはGitHub Actionsによる自動デプロイ（CI/CD）の導入も検討する。

| テーブル名                | 主要カラム                                                                               | 型                                              | 備考                                |
| -------------------- | ----------------------------------------------------------------------------------- | ---------------------------------------------- | --------------------------------- |
| employees            | employee_id, name, role                                                             | VARCHAR, VARCHAR, ENUM                         | role: admin / user                |
| drinks               | drink_id, name, image_url, stock, unit_weight                                       | SERIAL, VARCHAR, TEXT, INT, DECIMAL            | unit_weight は将来の重量センサー連携用（NULL許容） |
| transactions         | id, employee_id, drink_id, quantity, type, customer_name, source, created_at        | SERIAL, FK, FK, INT, ENUM, VARCHAR, ENUM, TIMESTAMP | type: takeout / return, source: manual / sensor（将来拡張用） |
| inventory_checks     | id, employee_id, drink_id, system_stock, actual_stock, diff, check_type, created_at | SERIAL, FK, FK, INT, INT, INT, ENUM, TIMESTAMP | check_type: manual / auto（将来拡張用）  |

| settings             | key, value                                                                          | VARCHAR, TEXT                                  | 通知先、棚卸しスケジュール等                    |
| 【将来】sensor_readings | id, sensor_id, drink_id, weight, estimated_count, created_at                        | SERIAL, VARCHAR, FK, DECIMAL, INT, TIMESTAMP   | 重量センサーの生データ                       |
| 【将来】lock_logs       | id, timestamp, mode, idm_hash, employee_id, unlock_permitted, transaction_id        | SERIAL, TIMESTAMP, TEXT, TEXT, FK, BOOLEAN, FK  | 電子錠解錠ログ（[詳細](./requirements-lock.md#5-データモデル)） |
| 【将来】registered_cards | id, idm_hash, employee_id, is_active, created_at                                    | SERIAL, TEXT, FK, BOOLEAN, TIMESTAMP            | 登録制限モード用カード（[詳細](./requirements-lock.md#5-データモデル)） |

### 6.1 将来拡張を考慮した設計指針

- `transactions` テーブルの `source` カラムにより、手動記録とセンサー自動記録を区別可能にする
- `inventory_checks` テーブルの `check_type` カラムにより、手動棚卸しと自動棚卸しを区別可能にする
- `drinks` テーブルの `unit_weight` に1本あたりの重量を登録しておくことで、重量センサー導入時に在庫数の自動算出が可能になる
- APIはRESTfulに設計し、将来のセンサーデータ受信用エンドポイント（`POST /api/sensor/readings`）を追加しやすい構造にする

## 7. 運用フロー

### 7.1 日常運用（ドリンク取り出し）

1. スタッフが冷蔵庫前のタブレットでアプリを開く
1. 社員番号を入力してログイン
1. ドリンク種類・数量を選択し「確定」を押下
1. 記録完了画面が表示される
1. 冷蔵庫からドリンクを取り出し、お客様に提供する

### 7.2 ドリンク返却

1. スタッフが冷蔵庫前のタブレットでアプリを開く
1. 社員番号を入力してログイン
1. 「返却」モードに切り替える（画面がオレンジ色テーマに変わる）
1. 返却するドリンク種類・数量を選択し「返却する」を押下
1. 返却完了画面が表示される
1. 冷蔵庫にドリンクを戻す

### 7.3 棚卸し運用（1日2回推奨）

1. 管理者またはスタッフが棚卸し画面を開く
1. 各ドリンク種類の実在庫数を入力
1. システムが理論在庫との差分を自動計算
1. 差分がある場合はアラートが管理者に通知される
1. 差分の原因を調査し、必要に応じて是正措置を実施

### 7.4 入庫運用

1. 管理者が入庫登録画面を開く
1. 補充するドリンク種類・数量を入力
1. 在庫数が自動更新される

### 7.5 【将来対応】FeliCa電子錠との連携

- FeliCaリーダー付き電子錠を冷蔵庫に導入し、本アプリと連携予定
- アプリで記録完了後にFeliCaタッチで解錠を許可する仕組みを構築
- 3モード運用（オープン/調査/登録制限）で段階的なセキュリティ強化が可能
- 解錠ログとアプリの取り出し記録を突合し、不整合を検知
- 詳細は [FeliCa電子錠連携 要件定義書](./requirements-lock.md) を参照

### 7.6 【将来対応】重量センサーによる自動在庫管理

- 冷蔵庫内の各棚に重量センサーを設置し、ドリンクの重量をリアルタイム計測
- 1本あたりの重量（`unit_weight`）をマスタに登録し、総重量から在庫数を自動算出
- 手動棚卸しの代替、または手動棚卸しとのクロスチェックとして活用
- 急激な重量変化（大量持ち出し等）を検知した場合、管理者に即時アラート

## 8. 盗難・無断持ち出し対策

本システムでは、以下の多層的なアプローチで無断持ち出しを抑止する。

### 8.1 システム的対策（本アプリで実現）

- 冷蔵庫前のタブレットで記録を取る動線設計により、入れ忘れを抑止
- 棚卸し機能による差分検知で、不正な持ち出しを早期発見
- 全取り出し記録のログにより、追跡可能性を確保
- 差分アラートにより、管理者が即座に状況を把握可能

### 8.2 物理的対策（別途実施）

- 【将来対応】ICカード対応電子錠を導入し、記録と連動した開錠制御を実現
- 冷蔵庫付近への小型カメラ設置（設置していることを明示し、抑止効果を狙う）
- タブレットの固定設置（盗難・移動防止）

### 8.3 【将来対応】センサーによる自動検知

- 重量センサーにより、記録なしの持ち出しをリアルタイムで検知
- アプリの取り出し記録とセンサーの重量変化を突合し、未記録の持ち出しを自動検出
- ICカードログ＋重量センサー＋アプリ記録の三重チェックにより、高精度な不正検知を実現

## 9. 将来拡張要件

本システムは段階的に拡張していく計画とする。各フェーズの概要を以下に示す。

### Phase 1（本開発スコープ）

ブラウザベースの在庫管理アプリを開発・導入する。

- 手動による取り出し記録・入庫記録・棚卸し
- 在庫ダッシュボード・差分アラート
- 社員番号認証（パスワード不要）・権限管理

### Phase 2：FeliCa電子錠連携

FeliCaリーダー付き電子錠を冷蔵庫に導入し、アプリと連携する。

- 対象機能：F-03（ICカード連携）、F-11（ICカード認証ログ）
- アプリで記録完了 → FeliCaタッチで解錠許可のフローを構築
- 3モード運用：オープンモード（通常）、調査モード（不正検知時）、登録制限モード（エスカレーション）
- Raspberry Pi等のコントローラーを介した制御、REST APIによる通信
- 前提：電子錠の到着、電源・ネットワーク環境の整備
- **詳細要件：[FeliCa電子錠連携 要件定義書](./requirements-lock.md) を参照**

### Phase 3：重量センサー連携

冷蔵庫内に重量センサーを設置し、在庫の自動計測を実現する。

- 対象機能：F-13（重量センサー連携）、F-14（自動棚卸し）、F-15（異常検知アラート）
- ドリンク種類ごとの重量をマスタ登録し、センサーデータから在庫数を自動算出
- 手動棚卸しの省力化・廃止
- 未記録の持ち出しをリアルタイム検知
- 前提：重量センサーの選定（精度・耐久性・通信方式）、中継デバイス（Raspberry Pi等）の設置

### 拡張時の技術的考慮事項

- Next.js API RoutesはRESTfulに設計し、新規エンドポイント（`POST /api/sensor/readings` 等）の追加が容易な構造にすること
- Prisma ORMを使用し、スキーマ変更は `prisma migrate` で管理。SQLite → PostgreSQL（RDS）への移行はPrismaの接続先変更のみで対応可能
- センサーデータの受信はMQTTまたはWebSocketに対応できるよう、通信レイヤーを抽象化すること（Node.jsの `mqtt` / `ws` パッケージを活用）
- データベースの `transactions` テーブルに `source` カラムを初期段階から用意し、将来の自動記録と手動記録の区別に備えること
- フロントエンドのダッシュボードは、センサーデータの表示ウィジェットを後から追加できるReactコンポーネント設計にすること
- スケールアウトが必要になった場合、Lightsail → ECS（Fargate）+ RDS への移行パスを想定

## 11. 前提条件・制約事項

### 11.1 前提条件

- 冷蔵庫前にタブレット端末を固定設置可能であること
- タブレット端末からWi-Fi接続が可能であること
- 8桁の社員番号が全スタッフに付与されていること
- 【将来対応に向けて】ICカード対応電子錠の設置場所・電源を確保可能であること
- 【将来対応に向けて】重量センサー設置のための冷蔵庫内スペースおよび電源・通信環境を確保可能であること

### 11.2 制約事項

- ICカード対応電子錠の導入までは、記録なしでの物理的な取り出しを完全に防止できない
- 物理的な破壊行為への対策は本システムのスコープ外とする
- ICカード対応電子錠の選定・導入・アプリ連携は将来フェーズで対応する
- 重量センサーの精度はドリンクの種類・容器の重量差に依存するため、導入時にキャリブレーションが必要
- 重量センサー連携の詳細仕様はセンサー選定後に別途定義する
